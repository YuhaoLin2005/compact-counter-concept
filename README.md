## 一、痛点：AI 黑盒焦虑

### 用户的真实体感

> "聊了一个小时，感觉 AI 开始胡说八道了... 但我不知道是真的在胡扯，还是我自己想多了。"

每个深度使用 LLM CLI 工具（Claude Code、DeepSeek-TUI、Aider）的开发者都经历过这种时刻：

- 对话越聊越长，回复质量在悄悄下降
- 模型开始遗漏你半小时前提到的关键约束
- 编造出你从未定义过的函数名、文件路径
- 你**隐约感觉**该重置了，但又不知道现在重置是否浪费

**这就是"黑盒焦虑"：你依赖一个不透明的系统做决策，却无法评估它的当前状态。**

### 现有工具的缺失

| 工具 | 告诉你上下文用量 | 告诉你压缩次数 | 告诉你质量风险 |
|------|:---:|:---:|:---:|
| ChatGPT Web | ✅ token 百分比 | ❌ | ❌ |
| Claude Code | ❌ | ❌ | ❌ |
| Cursor | ❌ | ❌ | ❌ |
| DeepSeek-TUI | ❌ | ❌ | ❌ |
| Aider | ❌ 地图模式有 | ❌ | ❌ |

**所有主流工具都缺失了同一个东西：上下文健康度指标。**

## 二、洞察：压缩次数 = 健康反向指标

### 为什么是"压缩次数"？

上下文窗口满了 → 系统自动压缩（把早期对话总结成简短摘要）→ 信息丢失 → 再来一轮 → 再丢 → ...

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ 原始对话  │ ──▶ │ 压缩 #1  │ ──▶ │ 压缩 #2  │ ──▶ │ 压缩 #4  │
│ 100% 信息 │     │ ~80% 信息 │     │ ~60% 信息 │     │ ~25% 信息 │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
     🟢               🟢               🟡               🔴
```

**压缩次数是一个天然的"反向健康指标"**：

- 它**必然发生**——只要对话够长，压缩一定会触发
- 它**单向累积**——不会自动减少，只增不减
- 它**与质量负相关**——次数越多，模型掌握的有效信息越少
- 它**客观可测**——日志中有关键词可以匹配

> **我的洞察：不需要让用户看懂压缩算法，只需要给 ta 一个数字 + 一种颜色。**

## 三、概念设计：让黑盒变白盒

### 三态对比 — 同一个终端界面的三种状态

```
┌─────────────────────────┐ ┌─────────────────────────┐ ┌─────────────────────────┐
│  🟢 健康状态            │ │  🟡 轻度警告            │ │  🔴 严重警告            │
│  ───────────────────────│ │  ───────────────────────│ │  ───────────────────────│
│  🤖 Opus 4.6            │ │  🤖 Opus 4.6            │ │  🤖 Opus 4.6            │
│  🔄 压缩: 0 次          │ │  🔄 压缩: 2 次          │ │  🔄 压缩: 4 次          │
│  📊 上下文: 23%         │ │  📊 上下文: 68%         │ │  📊 上下文: 91%         │
│  💰 $0.05               │ │  💰 $0.18               │ │  💰 $0.31               │
│  ───────────────────────│ │  ───────────────────────│ │  ───────────────────────│
│  ✅ 完整记忆            │ │  ⚠️ 部分信息已丢失      │ │  💀 高度幻觉            │
│  模型完全掌握           │ │  建议留意回答质量       │ │  强烈建议立即重置！     │
│  所有上下文。           │ │  可继续但不推荐         │ │  执行 /reset 或 /rewind │
│                         │ │  新增复杂任务。         │ │  到早期节点。           │
└─────────────────────────┘ └─────────────────────────┘ └─────────────────────────┘
```

### 设计逻辑

| 设计决策 | 理由 |
|----------|------|
| **只暴露 3 个指标** | 不让用户读仪表盘，只给最关键的信号 |
| **颜色 + 文字双通道** | 扫一眼（颜色）和细读（文字）两种消费模式 |
| **操作建议写在状态栏里** | 不把认知负担丢给用户，直接告诉 ta 该做什么 |
| **不追求绝对精度** | "~60% 上下文保持"比"精确 63.2%"更有用；产品指标要的是方向感，不是仪表精度 |
| **放在终端状态栏** | 不额外占用屏幕空间，融入已有心智模型 |

## 四、验证路径：设计 x 叙事 x 传播

代码实现不是第一步。验证需求是否真实存在，才是。

### Phase 1 — 概念验证（本仓库）

- [x] 用 ASCII 图让抽象概念可视化
- [x] 写清楚"为什么是压缩次数"的推导链
- [x] 提供可运行的监控脚本框架（证明技术可行）
- [ ] 收集第一轮社区反馈

### Phase 2 — 开源社区验证

在 Claude Code / Aider / DeepSeek 社区发起讨论：

> "如果终端状态栏显示'压缩次数'，你会用它来决定何时重置会话吗？"

- 在 GitHub Discussions / Discord / Reddit 发帖
- 收集用户对这个指标的直觉反应
- 统计"会用它" vs "无所谓"的比例

### Phase 3 — 迭代或转向

| 官方/社区反馈 | 后续动作 |
|--------------|---------|
| "确实有这个需求，我们已经在做了" | 需求验证通过 ✅，关闭提案 |
| "有意思，但优先级不高" | 收集具体顾虑，迭代设计 |
| "不考虑" | 问清楚原因，更新提案或转向 |
| 社区高票支持但官方沉默 | 考虑独立实现或 PR 贡献 |
| 无人关心 | **这也是发现**——记录为什么用户不关心，更新对市场的理解 |

## 六、为什么这个指标值得存在？

### 对用户
- **一个明确的信号**：什么时候该重置，不再靠猜
- **降低认知负载**：不用时刻担心"AI 是不是在胡扯"
- **节省费用**：及时重置可以避免在低质量上下文中浪费 token

### 对工具团队
- **一个零成本的 UX 提升**：监控日志不需要额外 API 调用
- **减少用户流失**：用户因为"AI 越来越笨"而放弃使用时，至少知道为什么
- **差异化竞争力**：目前没有竞品做这件事

### 对行业
- **把"可观测性"引入 AI 交互**：就像 DevOps 需要监控一样，AI-native workflow 也需要健康检查
- **从"魔法黑盒"到"可管理的工具"**：让用户从被动接受到主动掌控
