# 架构思路

> 这不是技术架构文档，是设计决策记录。解释"为什么这样设计"而不是"代码怎么写的"。

## 核心问题

**TUI 状态栏上的压缩次数应该怎么展示？**

这不是一个 UI 问题，是一个信息架构问题：
- 用户需要多少信息才能做出"该不该重置"的决定？
- 每多一个指标，就多一分认知负载
- 每少一个指标，就多一分不确定性

## 信息架构

### 选择暴露三个指标

```
🔄 压缩: 2 次     — 健康指标本体
📊 上下文: 68%    — 未来风险预警（离下次压缩还有多远）
💰 $0.18         — 行为后果可视化（压缩让成本在累积）
```

**取舍逻辑：**

| 考虑过的指标 | 是否纳入 | 理由 |
|-------------|:------:|------|
| 压缩次数 | ✅ | 核心指标，不能少 |
| 上下文使用率 | ✅ | 让用户有"离下次压缩"的预期 |
| 累计费用 | ✅ | 让用户感知"不重置的代价" |
| 距上次压缩时间 | ❌ | 不够直观，用户不好判断 |
| 压缩丢失的 token 数 | ❌ | 技术指标，用户不需要知道 |
| 剩余上下文 token 数 | ❌ | 和百分比功能重复 |
| 会话总时长 | ❌ | 和"健康度"没有直接关系 |

### 行为引导文字放在底部

状态的底部警告语是**可执行建议**，不是描述：

```
🟢 "模型完全掌握所有上下文" — 告诉用户：放心用
🟡 "建议留意回答质量，不推荐新增复杂任务" — 告诉用户：注意，但还能用
🔴 "强烈建议立即重置！执行 /reset"      — 告诉用户：别犹豫了，动手
```

**原则：不让用户做二次判断。** 如果你告诉用户"压缩2次"，ta 还要自己判断"2次严不严重？"——那你的设计是失败的。

## 监控脚本框架的设计取舍

`monitor.py` 的目的是**沟通意图**，不是上线运行。

故意保留的"不完整"：
- 没有处理所有边界情况（那是生产代码的事）
- 没有接入真实的通知系统（用框架 stub 代替）
- YAML 配置只在 README 中示例（没有引入 pyyaml 依赖）

**如果这个提案被认可，实现是简单的。如果提案本身不成立，写再多代码也没用。**

## 状态阈值的设定

```
0-1 次 → 🟢 HEALTHY
2-3 次 → 🟡 WARNING
4+ 次 → 🔴 CRITICAL
```

这是基于经验观察的初步设定，不是数据分析结论。一个有说服力的改进方向是：
1. 收集真实用户的压缩频率数据
2. 分析压缩次数与"用户手动重置"行为的相关性
3. 用数据调整阈值而不是拍脑袋

> **但作为概念验证阶段，允许合理假设。产品设计的艺术就是在数据不足时也能做出合理决策。**
